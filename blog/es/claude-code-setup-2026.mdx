## Context & Background

### The Evolution of AI Coding Tools

The AI-assisted coding timeline has three distinct phases. **Phase 1 (2021–2023)** was autocomplete: GitHub Copilot, Tabnine, and similar tools predicted the next few lines of code. Useful, but fundamentally passive. **Phase 2 (2023–2025)** was chat-in-IDE: tools like Cursor, Cody, and the original Claude Code let you ask questions about your codebase and generate multi-line implementations. Better, but still reactive — you prompt, it responds, you paste.

**Phase 3 (2025–present)** is **agentic coding**: the tool reads your codebase autonomously, plans multi-step implementations, executes them across files, runs tests, interprets failures, iterates, and commits — all with minimal human intervention. Claude Code in 2026 is firmly in Phase 3.

### What Makes Claude Code Different

Claude Code is a **terminal-native, long-running Node.js process** that connects directly to Anthropic's models (or local models via compatible endpoints). Unlike IDE-embedded tools, it operates at the filesystem and shell level — it can read files, write files, run arbitrary shell commands, manage git operations, and invoke external tools through MCP (Model Context Protocol) servers.

The key architectural concepts you need to understand:

**CLAUDE.md** — A Markdown file in your project root that acts as persistent memory. Claude reads it at session start and uses it for all architectural decisions. Think of it as a `README` written specifically for your AI agent.

**Hooks** — User-defined handlers that execute automatically at fixed points in Claude Code's lifecycle: before tool use, after tool use, on session start/stop, before compaction, and more. Four types: command (shell), HTTP, prompt (AI evaluation), and agent (subagent with tool access).

**Subagents** — Specialized AI agents that Claude delegates work to. Each has its own system prompt, tool restrictions, and optionally a different model. They run in isolated context, preventing long tasks from polluting your main conversation.

**Skills** — Workflow definitions that Claude activates automatically when relevant to the current task. Unlike slash commands (which you trigger manually), skills are proactive.

**Plugins** — Distributable packages that bundle commands, agents, skills, hooks, and MCP configs. Install with one command. The marketplace has over 9,000 entries as of March 2026.

**MCP Servers** — External tool integrations (GitHub, Supabase, Vercel, databases, browsers) that give Claude capabilities beyond the local filesystem.

## Deep Technical Analysis

### The CLAUDE.md System: Your Agent's Memory

The single most impactful configuration in Claude Code is the **CLAUDE.md** file. When Claude starts a session, it parses this file first, using it as persistent context for every decision.

CLAUDE.md files cascade from three levels:

1. **Project-level** (`.claude/CLAUDE.md` or `CLAUDE.md` in repo root) — Shared with your team via git. Highest priority for project-specific context.
2. **User-level** (`~/.claude/CLAUDE.md`) — Personal preferences, coding style, model selection rules. Applies to all projects.
3. **Directory-level** — CLAUDE.md files in subdirectories provide context when Claude is working in that area.

Here's a production-grade example adapted from the `everything-claude-code` repo:

```markdown
# Project: SaaS Dashboard

## Tech Stack
- Next.js 15 (App Router), TypeScript strict mode
- Tailwind CSS + shadcn/ui components
- Supabase (PostgreSQL + Auth + Realtime)
- Vercel deployment

## Architecture Rules
- All API routes use Route Handlers in app/api/
- Server Components by default; use "use client" only when necessary
- Database access through typed Supabase client (lib/supabase/server.ts)
- No raw SQL — use Supabase query builder

## Coding Conventions
- Functional components only, no class components
- Prefer named exports over default exports
- Use Zod schemas for all input validation
- Error boundaries at route layout level

## Commands
- `pnpm dev` — Start development server
- `pnpm test` — Run Vitest test suite
- `pnpm lint` — ESLint + Prettier check
- `pnpm db:migrate` — Run Supabase migrations

## Security Rules
- Never commit .env files or API keys
- All user input must be validated with Zod before database operations
- Use Supabase RLS policies — never bypass them
- Require approval before any destructive git operations
```

### Hooks: The Four Types

Hooks are the deterministic layer in Claude Code — they **always** run when configured, regardless of model behavior. This is what separates Claude Code from a chatbot: you can enforce invariants.

The hook lifecycle has 12 event types, but the most practically important are:

**PreToolUse** — Fires before Claude executes a tool (Read, Edit, Bash, etc.). You can block the action, modify it, or inject context. This is where you enforce security policies.

**PostToolUse** — Fires after a tool executes. Use for audit logging, output validation, or triggering downstream actions.

**Stop** — Fires when Claude finishes responding. You can force it to continue (preventing premature completion) or inject final validation.

**Notification** — Fires on permission prompts, idle prompts, and auth events.

Each event supports four handler types:

```
┌─────────────┬────────────────────────────────────────────┐
│ Type        │ How it Works                               │
├─────────────┼────────────────────────────────────────────┤
│ command     │ Runs a shell script, reads exit code + JSON│
│ http        │ POSTs JSON to a URL, reads JSON response   │
│ prompt      │ Sends context to a Claude model (Haiku)    │
│             │ for a single-turn yes/no decision           │
│ agent       │ Spawns a subagent (up to 50 tool-use turns)│
│             │ that can inspect the codebase before        │
│             │ returning a decision                        │
└─────────────┴────────────────────────────────────────────┘
```

Here's a practical PreToolUse hook that prevents commits to main/master:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node -e \"const input = JSON.parse(require('fs').readFileSync('/dev/stdin','utf8')); const cmd = input.tool_input?.command || ''; if (/git\\s+(push|commit)/.test(cmd) && /(\\bmain\\b|\\bmaster\\b)/.test(cmd)) { process.stderr.write('BLOCKED: Direct commits to main/master are not allowed. Create a feature branch.'); process.exit(2); }\""
          }
        ]
      }
    ]
  }
}
```

Exit code semantics: `0` = approve, `2` = block (stderr shown as reason), any other = non-blocking error.

### Subagents and Worktree Isolation

Subagents are where Claude Code becomes a true orchestration framework. Instead of running everything in a single, ever-growing context, you delegate specialized work to focused agents.

A subagent definition is a Markdown file with YAML frontmatter:

```yaml
---
name: security-reviewer
description: Analyzes code for security vulnerabilities, injection risks, and auth issues
tools:
  - Read
  - Grep
  - Glob
  - Bash
model: opus
isolation: worktree
---

You are a senior security engineer. Your job is to identify vulnerabilities,
not fix them. Report findings using this severity format:

🔴 CRITICAL - Exploitable in production, data loss possible
🟠 HIGH - Significant risk, should fix before merge
🟡 MEDIUM - Defense-in-depth issue, fix within sprint
🔵 LOW - Best practice recommendation

For each finding, provide:
1. File path and line number
2. Vulnerability type (OWASP category)
3. Proof of concept or attack scenario
4. Recommended fix
```

The `isolation: worktree` directive is the key 2026 addition. When this subagent runs, Claude creates a temporary git worktree, executes the review in complete isolation, then destroys the worktree on completion. The main session's working directory is never touched. This enables a powerful pattern: **spawn multiple worktree-isolated subagents in parallel**, each working on a different aspect of the codebase.

### The MCP Integration Layer

MCP (Model Context Protocol) servers extend Claude Code's capabilities beyond the local filesystem. Each server exposes tools that Claude can invoke during a session.

A critical operational concern from the `everything-claude-code` repo: **don't enable all MCPs at once**. Each server's tool definitions are loaded into your context window at session start. With 20+ MCP servers active, your 200K context window can shrink to 70K before you even send your first message.

The recommended approach from the community:

- Have 20–30 MCPs **configured** in your `~/.claude.json`
- Keep **under 10 enabled** per project
- Keep **under 80 total tools** active
- Use `disabledMcpServers` in project config to selectively disable unused ones

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "YOUR_TOKEN" }
    },
    "supabase": {
      "command": "npx",
      "args": ["-y", "@supabase/mcp-server-supabase@latest"],
      "env": { "SUPABASE_ACCESS_TOKEN": "YOUR_TOKEN" }
    }
  }
}
```

## Hands-On: Production Setup from Scratch

### Prerequisites

- **Node.js 18+** (LTS recommended)
- A **Claude Pro, Max, or API** subscription
- **Git** configured with your credentials
- A code project to work with

### Step 1: Installation

The native CLI installer is the recommended path for macOS and Linux:

```bash
# Native installer — handles binary paths and auto-updates
curl -fsSL https://claude.ai/install.sh | bash

# Verify installation
claude --version
# Should show v2.1.x or later (required for plugins)
```

For npm environments (or Windows via WSL2):

```bash
npm install -g @anthropic-ai/claude-code
```

### Step 2: Authentication and Initial Configuration

```bash
cd your-project
claude

# You'll be prompted to authenticate:
# Option A: Claude Pro/Max subscription (browser OAuth)
# Option B: API key from platform.claude.com
```

Immediately after authentication, harden your security config:

```bash
# Edit the global settings
nano ~/.claude/settings.json
```

```json
{
  "permissions": {
    "deny": [
      "Bash(docker rm*)",
      "Bash(rm -rf /)",
      "Bash(git push origin main)",
      "Bash(git push origin master)"
    ]
  }
}
```

### Step 3: Install the everything-claude-code Plugin

The `everything-claude-code` repo (35,600 stars, MIT licensed) by Anthropic hackathon winner Affaan Mustafa is the most comprehensive community configuration collection. Install it as a plugin:

```bash
# Inside Claude Code, run:
/plugin marketplace add affaan-m/everything-claude-code

# Install the plugin
/plugin install everything-claude-code@everything-claude-code
```

This gives you immediate access to all slash commands (`/tdd`, `/plan`, `/code-review`, `/build-fix`, `/e2e`, `/verify`, `/learn`, and more), specialized subagents (planner, architect, security-reviewer, code-reviewer, TDD guide, build-error-resolver, e2e-runner, doc-updater, Go reviewer), and the continuous learning system.

**Important:** Rules can't be distributed via plugins (upstream limitation). Install them manually:

```bash
git clone https://github.com/affaan-m/everything-claude-code.git
cp -r everything-claude-code/rules/* ~/.claude/rules/
```

### Step 4: Create Your CLAUDE.md

In your project root, create a CLAUDE.md tailored to your stack:

```bash
cat &gt; CLAUDE.md &lt;&lt; 'EOF'
# Project Context

## Stack
[Your tech stack here]

## Commands
- `[your dev command]` — Start dev server
- `[your test command]` — Run tests

## Conventions
- [Your coding style rules]
- [Your file organization rules]

## Security
- Never commit secrets
- Validate all inputs
- Require approval for destructive operations
EOF
```

### Step 5: Configure Hooks for Memory Persistence

The `everything-claude-code` repo includes hooks that automatically save and load context across sessions. The hooks are in `hooks/hooks.json` and auto-load when the plugin is installed. The memory persistence system uses three scripts:

- **session-start.js** — Loads previous session context on startup
- **session-end.js** — Saves current state when the session terminates
- **pre-compact.js** — Preserves state before context compaction (when the conversation gets too long)

### Step 6: Verify and Test

```bash
# Inside Claude Code:
/cost                    # Check token usage
/model                   # See active model
claude agents            # List all configured agents
/plan "Add user auth"    # Test plan mode
```

The lesson (documented in the `everything-claude-code` repo): keep under 10 MCPs enabled per project and under 80 total tools active.

## Security & Risk Considerations

### Default Permissions Are Dangerously Broad

Out of the box, Claude Code has highly permissive execution rights. It can run arbitrary shell commands, write to any file in your project, and execute git operations. Your first action after installation should be configuring `~/.claude/settings.json` to restrict dangerous operations.

### Credential Exposure

Claude Code reads your environment variables and can access any credentials available in your shell session. If you're using MCP servers, API keys are stored in `~/.claude.json` — sometimes in plaintext. Use environment variables with a secrets manager rather than hardcoding tokens in config files.

### Plugin Supply Chain Risk

The plugin marketplace has over 9,000 entries with no mandatory code signing or security review. Before installing any plugin, review its source on GitHub. This mirrors the ClawHub supply chain risks documented in the OpenClaw ecosystem — third-party agent extensions are fundamentally untrusted code execution.

### Hardening Checklist

1. **Restrict destructive commands** in `settings.json` — deny `docker rm`, `rm -rf`, direct pushes to main/production branches
2. **Set API spending limits** — both in the `settings.json` and at the provider level on platform.claude.com
3. **Use PreToolUse hooks** to validate Bash commands before execution. The agent-type hook is particularly powerful: spawn a security subagent that inspects the proposed command before approving it
4. **Never run Claude Code as root** or with elevated privileges
5. **Use `CLAUDE_CODE_SIMPLE` mode** for untrusted projects — this disables MCP tools, attachments, hooks, and CLAUDE.md loading for a minimal attack surface
6. **Audit MCP server permissions** — each server gets the tool access you grant it. Restrict to the minimum necessary

### Compliance Considerations

For GDPR/NIS2 compliance: when using Claude Code with Anthropic's cloud models, your code and conversation context is transmitted to Anthropic's servers. For sensitive codebases, use the API with data residency controls (the `inference_geo` parameter lets you specify US-only inference at 1.1x pricing) or route through a local model via Ollama for zero-egress operation. Anthropic now offers HIPAA-ready Enterprise plans for organizations processing protected health information.

## Key Takeaways & Actionable Summary

- **Create a CLAUDE.md file immediately** if you haven't already. This single file has more impact on Claude Code's usefulness than any other configuration. Include your stack, conventions, commands, and security rules.

- **Install the everything-claude-code plugin** (`/plugin marketplace add affaan-m/everything-claude-code`). It gives you battle-tested agents, hooks, skills, and commands from an Anthropic hackathon winner. 35,600 stars for a reason.

- **Use subagents for specialized work.** Don't run everything in your main context. Delegate code review, security analysis, testing, and documentation to purpose-built agents with `isolation: worktree` for clean parallel execution.

- **Manage your MCP server budget.** Keep under 10 enabled per project, under 80 tools active. Use `disabledMcpServers` aggressively. Your context window is finite.

- **Harden your security config on day one.** Deny destructive bash commands. Set API spending limits. Use PreToolUse hooks for validation. Never run as root.

- **Use Plan Mode (`/plan` or Shift+Tab) before complex tasks.** Let the agent analyze the codebase and produce an execution plan you review before any files change.

- **Use Remote Control for long tasks.** With a Max subscription (v2.1.52+), start a build, walk away, and monitor/approve from the Claude mobile app.

- **Pick the right model for the task:** Opus 4.6 for architecture decisions and complex reasoning, Sonnet 4.6 for everyday coding (default for most work), Haiku 4.5 for quick checks and lightweight reviews.
